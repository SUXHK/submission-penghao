# 需求分析说明

## 1. 问题定义

在真实软件交付中，FSE（Field Service Engineer / Full-Stack Engineer）处理缺陷不仅仅是把状态从"待修复"改成"已修复"。需要完成：问题现象重现 → 影响范围评估 → 优先级判断 → 根因假设 → 修复计划 → 执行修复 → 验证回归 → 关闭沉淀 的完整工程闭环。

当前缺少一个小型化、可演示的系统，来帮助 FSE 管理缺陷从受理到验证关闭的全过程，同时体现 **AI 辅助分析但不替代人工判断** 的协作模式。

## 2. 目标与价值

| 目标 | 价值 | 衡量方式 |
|---|---|---|
| 缺陷全生命周期管理 | 提供从登记到关闭的结构化流程，避免信息遗漏 | 每个缺陷可走完整状态流转，状态不可跳跃 |
| AI 辅助排查与建议 | 利用 DeepSeek 生成排查路径、根因假设、测试建议，提升效率 | AI 生成建议后人工审核，记录采纳/拒绝率 |
| 人工判断保障决策质量 | 优先级、严重程度、关闭结论由人确认，AI 仅提供参考 | 所有关键决策节点需要人工确认操作 |
| 知识沉淀可复用 | 缺陷关闭后转化为回归用例、排查手册、风险规则 | 知识库可搜索、可关联回原缺陷 |
| 端到端可演示 | 提交→修复→验证→关闭→复盘，全链路可运行 | 评委可按预设账号走完完整流程 |

## 3. 用户场景

| 场景 | 用户 | 当前痛点 | 期望结果 |
|---|---|---|---|
| 缺陷登记与补齐 | 提交人(Submitter) | 缺陷信息不全就进入修复，导致反复沟通 | 系统校验必填字段（现象/环境/复现步骤/期望结果/实际结果），不全则阻止流转 |
| 分诊与优先级判断 | 工程师(Engineer) | 优先级凭感觉定，没有基于影响范围和业务影响 | 系统引导评估用户影响/业务影响/频率/规避方案/发布窗口，综合判定优先级 |
| 根因分析与修复计划 | 工程师(Engineer) | 修复无计划，直接改代码 | 记录根因假设、修复方案、影响范围，AI 辅助生成建议 |
| 修复执行与记录 | 工程师(Engineer) | 修复后无记录，无法追溯 | 记录修复内容、涉及文件/模块、耗时 |
| 验证与关闭 | QA | 修复完成直接关闭，无验证 | 必须有验证记录、回归范围、验证结论方可关闭 |
| 知识沉淀 | 全员 | 缺陷经验无法复用 | 关闭后生成回归用例/排查手册/风险规则，支持搜索 |

## 4. 范围定义

### 本次实现

- 缺陷登记表单（现象/环境/复现步骤/期望结果/实际结果/附件截图）
- 缺陷信息完整性校验（必填项不满足则不能进入修复）
- 严重程度评估维度（用户影响/业务影响/发生频率/规避方案/发布窗口）
- 优先级自动推荐 + 人工确认
- 9 状态完整流转（DRAFT → REPORTED → TRIAGING → ANALYZED → PLANNED → IN_REPAIR → FIXED → VERIFIED → CLOSED，含 REOPENED 共 10 种状态）
- 状态流转规则校验（前置条件不满足则不能流转）
- AI 辅助生成：排查路径建议、根因假设、修复计划建议、修复内容建议、测试场景建议、复盘草稿（6 种 AI 建议类型）
- AI 建议逐条审核（采纳/拒绝/修改），记录人工决策
- JWT 用户认证（3 角色：Submitter / Engineer / QA）
- 角色权限控制（不同角色可执行的操作不同）
- 缺陷附件上传（截图/日志，限制 5MB/文件，最多 5 个）
- 知识库模块（缺陷关闭后自动生成：回归用例、排查手册条目、风险规则）
- 知识库搜索与分类筛选
- 缺陷看板/列表页（按状态列展示，支持筛选和搜索）

### 本次不实现

- 真实日志系统接入（题目明确不要求）
- 邮件/消息通知
- 缺陷与代码仓库（Git）的关联
- 批量导入/导出
- 数据统计看板/仪表盘
- 移动端适配
- 国际化
- OAuth/SSO 集成

## 5. 业务规则与约束

### 5.1 缺陷状态流转表

```
DRAFT(草稿) → 可删除、编辑，不进入正式流程
  ↓ (提交)
REPORTED(已登记) → 信息补齐完毕，等待分诊
  ↓ (开始分诊)
TRIAGING(分诊中) → 评估严重程度和优先级，AI 生成排查建议
  ↓ (完成分诊)
ANALYZED(已分析) → 记录影响范围和根因假设，AI 生成根因假设
  ↓ (制定修复计划)
PLANNED(已计划) → 确认修复方案和责任人
  ↓ (开始修复)
IN_REPAIR(修复中) → 执行修复操作，记录修复内容
  ↓ (修复完成)
FIXED(已修复) → 等待验证，不可直接关闭
  ↓ (验证通过)          ↓ (验证不通过)
VERIFIED(已验证)       IN_REPAIR(修复中) [回退]
  ↓ (确认关闭)
CLOSED(已关闭) → 触发知识沉淀，不可再编辑
  ↓ (发现新问题)
REOPENED(重新打开) → 回到 REPORTED 状态重新走流程
```

### 5.2 关键校验规则

| 规则 | 说明 | 触发时机 | 处理方式 |
|---|---|---|---|
| 复现信息必填 | 现象、环境、复现步骤、期望结果、实际结果均为必填 | REPORTED 状态提交时 | 阻止提交，前端标红未填字段 |
| 影响评估先于优先级 | 未完成影响维度评估（用户影响/业务影响/频率/规避方案/发布窗口）不能确认优先级 | TRIAGING 状态确定优先级时 | 弹出未完成评估项，阻止流转 |
| 修复方案先于修复执行 | 未填写根因假设和修复方案不能进入 IN_REPAIR | PLANNED → IN_REPAIR 流转时 | 阻止流转，提示需要填写修复方案 |
| 验证先于关闭 | 无验证记录（验证人/验证结果/回归范围/验证结论）不能关闭 | FIXED → VERIFIED → CLOSED 流转时 | 阻止关闭，提示需要 QA 角色完成验证 |
| 角色权限边界 | Submitter 不能修复，Engineer 不能最终关闭，QA 不能修改修复方案 | 各操作点 | 后端返回 403，前端隐藏操作按钮 |
| AI 建议必须审核 | AI 生成的建议默认为"待审核"状态，工程师必须逐条确认 | AI 建议生成后 | 展示审核面板，未审核的建议标记为橙色提醒 |
| 附件限制 | 单文件 ≤ 5MB，每个缺陷最多 5 个附件，仅允许图片和文本文件 | 上传时 | 前端+后端双重校验，超限弹提示 |

### 5.3 严重程度与优先级规则

**严重程度判定维度（1-5分级）：**

| 维度 | 权重 | 1级 | 5级 |
|---|---|---|---|
| 用户影响 | 30% | 单个用户偶发 | 全量用户功能不可用 |
| 业务影响 | 30% | 仅视觉瑕疵 | 核心业务链路中断 |
| 发生频率 | 20% | 极难复现 | 每次必现 |
| 规避方案 | 10% | 有简单规避方法 | 无任何规避手段 |
| 发布时间窗口 | 10% | 无发布压力 | 版本发布前必须修复 |

**优先级自动计算：** P0(紧急) / P1(高) / P2(中) / P3(低) / P4(建议)，人工可覆盖自动计算结果。

## 7. 开放问题与人工确认

| 问题 | 影响 | 当前决策 | 后续处理 |
|---|---|---|---|
| DeepSeek 返回格式稳定性 | AI 建议的 JSON 格式可能不稳定 | 后端增加响应解析容错，解析失败则返回原始文本作为建议 | 测试阶段用多种缺陷场景验证 |
| 前后端开发联调方式 | 影响开发效率 | 开发阶段前端独立 Vite dev server(:3000)，后端 :8080，通过 Vite proxy 转发 | 若 CORS 问题，后端配置允许 localhost:3000 |
| 附件存储路径（Docker） | Docker 重启后文件可能丢失 | Docker Compose 挂载宿主机路径 | 本地开发时存 ./uploads 目录，Docker 挂载此目录 |
| 3 角色切换演示 | 答辩时需要切换账号演示不同操作 | 分别预置 3 个账号，浏览器多 Tab 或退出登录切换 | README 中列出预置账号密码 |

## 8. 验收标准

| 验收点 | 验证方式 | 目标 |
|---|---|---|
| 缺陷登记完整性校验 | 提交空表单，检查前端+后端阻止 | 必填项不全时无法提交 |
| 状态流转规则生效 | 尝试跳状态流转（如 REPORTED 直接到 FIXED） | 后端拒绝并返回错误 |
| AI 建议生成 | 缺陷进入分诊后，点击"请求 AI 建议" | DeepSeek 返回排查路径/根因假设/测试建议 |
| AI 建议审核 | 对 AI 建议点击采纳/拒绝/修改 | 记录保存，审核状态更新 |
| 验证先于关闭 | QA 角色未填写验证记录时尝试关闭 | 后端拒绝流转 |
| 角色权限控制 | Submitter 尝试执行修复操作 | 返回 403 |
| 知识沉淀生成 | 缺陷关闭后检查知识库 | 自动生成回归用例/排查手册/风险规则条目 |
| 登录认证 | 未登录访问 API | 返回 401 |
| 附件上传 | 上传 >5MB 文件 | 后端拒绝 |
| 端到端流程 | 用预置账号从登记走到关闭 | 全流程通过 |

## 9. 风险与依赖

| 风险 / 依赖 | 影响 | 应对措施 |
|---|---|---|
| DeepSeek API 不可用或超时 | AI 建议功能无法使用 | 后端设置 15s 超时，超时则返回提示"AI 服务暂时不可用"，不阻塞主流程 |
| 本机 Docker 服务未启动 | MySQL 无法连接，应用启动失败 | README 明确说明必须先启动 Docker，后端增加数据库连接失败时的友好提示 |
| 前端多状态视图开发量大 | 前端开发时间可能不足 | 优先实现核心状态（REPORTED → TRIAGING → IN_REPAIR → FIXED → VERIFIED → CLOSED），DRAFT 和 ANALYZED/PLANNED 简化 |
| Java 全栈知识不足 | 后端开发效率可能低 | 使用 AI Agent 辅助生成后端代码，重点关注 Spring Boot 基础 CRUD + 校验逻辑 |
| 2 天时间限制 | 功能范围需要取舍 | 严格按"本次实现/本次不实现"范围执行，不实现部分在已知问题中说明 |
