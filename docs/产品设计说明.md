# 产品设计说明

## 1. 产品定位

面向 FSE（Field Service Engineer）的缺陷分诊与修复闭环系统。帮助工程师管理缺陷从受理、分析、修复、验证到知识沉淀的端到端流程，通过 AI 辅助分析提升效率，但不替代人工判断。

**核心价值**：结构化缺陷处理流程，确保每一步有据可查，最终沉淀为团队可复用的知识。

## 2. 用户与角色

| 角色 | 目标 | 关注点 | 典型操作 |
|---|---|---|---|
| 提交人 (Submitter) | 准确登记缺陷 | 复现信息完整性 | 创建缺陷、填写复现步骤、提交分诊 |
| 工程师 (Engineer) | 高效分析修复 | 根因定位、修复方案质量 | 分诊评估、根因分析、修复执行、审核AI建议 |
| QA | 确保修复质量 | 验证充分性、回归范围 | 验证结果填写、回归测试、关闭缺陷 |

## 3. 核心场景

### 场景1：缺陷登记与提交
- 用户：Submitter
- 背景：测试或用户反馈了一个缺陷
- 目标：完整记录缺陷信息，提交分诊
- 关键流程：创建 → 填写6项必填信息 → 提交分诊
- 成功标准：必填信息不全时系统阻止流转

### 场景2：工程师分诊与修复
- 用户：Engineer
- 背景：缺陷已登记，等待分析
- 目标：评估影响、分析根因、制定计划、执行修复
- 关键流程：分诊中→评估维度→已分析→根因假设→已计划→修复方案→修复中→修复内容→已修复
- 成功标准：每步校验通过方可流转，AI建议可审核

### 场景3：QA验证与关闭
- 用户：QA
- 背景：缺陷已修复，需验证
- 目标：验证修复有效性，确认可关闭
- 关键流程：填写验证记录→已验证→确认关闭→已关闭
- 成功标准：无验证记录不可关闭，自动生成知识条目

## 4. 核心对象抽象

| 对象 | 说明 | 为什么重要 |
|---|---|---|
| 缺陷(Defect) | 核心业务对象，承载全部缺陷信息 | 贯穿全流程的数据载体 |
| 状态流转(StateTransition) | 每次状态变更的审计记录 | 可追溯，符合工程规范 |
| AI建议(AISuggestion) | AI生成的排查/分析/测试建议 | 人机协同的载体 |
| 知识条目(KnowledgeItem) | 缺陷关闭后的沉淀产物 | 经验复用，防止重复踩坑 |
| 附件(Attachment) | 截图/日志等复现证据 | 补全复现信息 |

## 5. 产品主流程

```
创建缺陷 → 填写复现信息 → 提交分诊
  → 分诊评估(5维度) → 优先级自动计算 → 已分析
  → 填写根因假设 → 已计划
  → 填写修复方案 → 修复中
  → 填写修复内容 → 已修复
  → 填写验证记录 → 已验证
  → 确认关闭 → 已关闭
  → 自动生成知识条目
```

## 6. 页面设计

| 页面 | 目标 | 主要信息 | 关键操作 |
|---|---|---|---|
| 登录/注册 | 身份认证 | 用户名、密码、角色 | 登录、注册 |
| 缺陷列表(首页) | 查看全部缺陷 | 统计图表、状态筛选、缺陷表格 | 创建缺陷、点击查看详情 |
| 缺陷详情(Sheet) | 查看和编辑缺陷 | 复现信息、评估维度、根因修复、验证记录、AI建议 | 状态流转、字段编辑、AI建议审核 |
| 知识库 | 查看沉淀知识 | 知识条目列表 | 搜索、筛选、发布 |

## 7. 关键设计取舍

- **Sheet弹窗 vs 独立页面**：选择Sheet从右侧滑出，保持列表上下文可见
- **客户端筛选 vs 服务端分页**：选择客户端筛选，100+数据量下体验更流畅
- **9状态流转 vs 简化状态**：选择完整9状态（含REOPENED共10种），体现工程严谨性
- **AI建议三态审核**：采纳/拒绝/修改，充分记录人工决策过程
- **知识自动生成 vs 手动创建**：关闭时自动生成3类知识条目，人工审核后发布
