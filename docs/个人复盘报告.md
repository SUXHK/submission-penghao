# 个人复盘报告

## 1. 本次交付结果概述

实现了一个可运行的缺陷分诊与修复闭环 Web 应用，包含：
- 9状态流转机（DRAFT→REPORTED→TRIAGING→ANALYZED→PLANNED→IN_REPAIR→FIXED→VERIFIED→CLOSED，含REOPENED共10种状态）
- 3角色权限隔离（Submitter/Engineer/QA）
- AI辅助建议（DeepSeek生成6种建议：排查路径/根因假设/修复计划/修复内容/测试建议/复盘草稿，人工三态审核）
- 知识沉淀（关闭后自动生成回归用例/排查手册/风险规则）
- 完整的前后端分离架构（React 19 + Spring Boot 3 + MySQL 8）
- 后端单元测试20个全通过，E2E测试16项全通过

## 2. 对 AI-Native SDLC 的理解变化

AI Coding 下的 SDLC 最大差异在于：**人从"写代码"变成"审代码"**。

传统开发中，编码占据了大部分时间。AI Coding 下，AI 可以一次性生成大量代码（本次后端 58 个 Java 文件），人的核心工作变成了：
1. **需求确认**：AI 无法替代业务理解，需要人做关键决策
2. **架构把控**：模块划分、接口设计需要人的经验
3. **代码审核**：AI 生成的代码 95% 可用，但 5% 的问题需要人的判断
4. **迭代调优**：AI 对 UI/UX 的理解较弱，需要多轮迭代

## 3. 对 FSE 的理解变化

FSE 不只是全栈编码，还应具备：
- **端到端思维**：从缺陷受理到知识沉淀的完整链路
- **规则设计能力**：状态机、校验规则、优先级算法的设计
- **人机协同判断**：知道什么时候用 AI，什么时候靠自己
- **交付闭环意识**：代码→测试→部署→文档，缺一不可

## 4. 做得好的地方

- 9状态流转规则设计完整，每个状态都有进入/退出条件
- AI建议三态审核（采纳/拒绝/修改）符合"人负责判断"原则
- 客户端筛选+TanStack Table排序，100+数据量下体验流畅
- Sheet面板设计保持列表上下文，交互连贯
- E2E测试覆盖全流程，可复现

## 5. 做得不好的地方

- 统计栏设计反复调整多次，最初没有对齐用户预期
- 前端内联编辑的输入框方案踩坑（defaultValue切换问题）
- Playwright E2E测试调试耗时较长
- 代码注释偏少，部分逻辑需要结合文档理解

## 6. AI 使用中的经验

| 经验 | 说明 | 后续如何复用 |
|---|---|---|
| 先设计再编码 | 需求→详细设计→任务划分→编码，链路清晰 | 所有项目沿用此流程 |
| 提问式确认 | 不确定时用选项提问，而非猜测 | 需求阶段标准做法 |
| E2E测试自动化 | Playwright脚本全流程验证 | 每次改动后运行验证 |
| 模块独立开发 | 每个模块可独立编译测试 | 降低模块间耦合 |

## 7. 失败案例与教训

| 问题 | 原因 | 下次改进 |
|---|---|---|
| UI反复迭代10+次 | AI对视觉设计理解弱 | 首轮提供参考截图或设计稿 |
| Sheet关闭按钮找不到 | ×字符编码问题 | 用语义化data属性代替特殊字符 |
| defaultValue状态丢失 | 切换字段时DOM复用 | 用key prop强制重建 |

## 8. 对团队 AI Coding 转型的建议

- 建立项目模板和代码规范，减少 AI 生成代码的风格差异
- 推行"需求→设计→任务→编码"四阶段流程，每个阶段有明确交付物
- E2E测试应该成为 AI Coding 的标配，确保生成代码的行为正确性
- 建立 AI 协作记录制度，沉淀可复用的 Prompt 和流程
